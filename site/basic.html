HTML:

  • HTML --> HyperText Markup Language
  • Hypertext --> TEXT THT CONTAINS LINKS TO OTHER TEXT
  • Markup --> MARK SOMETHING UP TO ANNOTATE, ITS ALL ABOUT CONTENT WRAPPED UP AROUND TAGS
  • HTML --> FOR STRUCTURE
  • HTML COMMUNICATES INSTRUCTIONS TO THE WEB BROWSER. INDICATES HOW THE PAGE SHOULD BE PRESENTED TO THE VIEWER.
  • HTML TAGS ARE CALLED TAG/ELEMENT
  • Browsers ALWAYS interpret HTML sequentially, top to bottom
  • quirks mode --> NON-COMPLIANT PAGES. WHEN THE BROWSER Assumes that the HTML in the web page is NOT following the HTML standard, i.e., not in standards mode. Styles won't work correctly, etc.
  • standard mode --> COMPLIANT PAGES
  • What would happen if you didn't specify the HTML5 declaration? The page will be interpreted in quirks mode
  • BLOCK LEVEL ELEMENTS: (OFFICIALLY NOT PART OF HTML5, BUT STILL USED QUITE OFTEN IN LITERATURE AND REGULAR CODING)
    • RENDER TO BEGIN ON A NEW LINE (BY DEFAULT)
    • MAY CONTAIN INLINE OR OTHER BLOCK-LEVEL ELEMENTS
    • ROUGHLY FLOW CONTENT (HTML5 CATEGORY)
  • INLINE ELEMENTS: (OFFICIALLY NOT PART OF HTML5, BUT STILL USED QUITE OFTEN IN LITERATURE AND REGULAR CODING)
    • RENDER ON THE SAME LINE (BY DEFAULT), BUT CAN BE CHANGED
    • MAY ONLY CONTAIN OTHER INLINE ELEMENTS
    • ROUGHLY PHRASING CONTENT (HTML5 CATEGORY)

HTML FORMS:

  • USED TO GATHER INPUT FROM UR USER
  • FORM DATA CAN BE INPUTTED INTO A DB OR SENT TO AN EMAIL ADDRESS OR SOME OTHER FORM OF DATA PROCESS
  • INPUT FORMS ARE CREATED IN HTML
  • DATA IS SUBMITTED TO A PHP SCRIPT FOR PROCESSING.....PHP SCRIPT VALIDATES THE FORM...VALIDATION CAN BE DONE IN CLIENT SIDE PROGRAMMING LANGUAGES SUCH AS JavaSscript AND ALSO IN SERVER SIDE LANGUAGES SUCH AS PHP....RECOMMENDED TO IMPLEMENT BOTH.....FYI, USERS CAN DISABLE CLIENT SIDE VALIDATION BY ADJUSTING SETTING IN WEB BROWSER
  • EXAMPLES OF DATA VALIDATION TYPES: EMAIL FORMAT/MIN OR MAX LENGTH/PHONE NUMBER FORMAT
  • HTTP PROTOCOL FACILITATES COMMUNICATION BETWEEN CLIENT(WEB Browser) AND SERVER. TWO TYPES OF DATA TRANSMISSION METHODS BETWEEN CLIENT & SERVER: POST & GET
  • POST vs GET:

      • GET:

    • REQUESTS DATA FROM SPECIFIED SOURCE
    • NAME/VALUE PAIRS ARE SENT IN THE URL STRING. EXAMPLE: send.php?fname=john&lname=smith
    • REQUESTS CAN BE CACHED
    • REMAIN IN BROWSER HISTORY
    • SHOULD ONLY BE USED TO RETRIEVE DATA
    • HAVE LENGTH RESTRICTIONS
    • SHOULD NOT BE USED FOR SENSITIVE DATA
    • CAN BE BOOKMARKED

      POST:

    • SUBMITS DATA TO BE PROCESSED TO A SPECIFIED SOURCE
    • SENDS NAME/VALUE PAIRS IN THE HTTP MESSAGE BODY.
    • EXAMPLE:
      • POST send.php HTTP/1.1
      • Host: youaccel.com
      • fname=john&lname=smith
    • NO RESTRICTIONS ON DATA LENGTH
    • CANNOT BE BOOKMARKED
    • DO NOT REMAIN IN BROWSER HISTORY
    • REQUESTS ARE NEVER CACHED
